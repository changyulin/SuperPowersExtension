# Retrieval Workflow

## 目标

在真正动手前，先快速找出“这个项目以前已经知道什么”，减少重复试错。

## 查询输入

优先使用下面这些词组成查询：

- 错误关键词
- 模块名
- 目录名
- 文件名
- 测试名
- 工具名
- 用户刚刚强调的偏好词

示例：

- `pytest env app init`
- `mcp tools timeout`
- `session store memory`

## 检索顺序

1. 搜项目内 `learnings.jsonl`
2. 如果命中条目里提到了相关文档，再读项目内 `solutions\` / `decisions\`
3. 再搜全局 `shared\learnings.jsonl`
4. 最后按需读全局 `shared\solutions\` / `shared\decisions\`

## 排序原则

按下面的顺序综合排序：

1. 是否来自当前项目
2. 查询词命中数量
3. 标题与规则字段命中
4. 标签与文件字段命中
5. 置信度
6. 更新时间

## 使用原则

- 默认展示前 3 到 5 条，不要把整库倒出来
- 项目内和全局冲突时，项目内优先
- 如果命中的经验只描述“现象”，不要机械照抄，先核对当前代码是否仍然一致
- 如果经验对应的文件已经不存在，把它视为可疑命中，后续加入清理候选

## 何时追加新经验

满足任一条件就应新增原子经验：

- 修复了一个明确错误，并知道为什么
- 同一做法已经重复出现两次以上
- 用户纠正了之前的默认做法
- 你刚做出了一个以后会影响别人修改方式的决策

## 何时升级为文档

满足任一条件就应补 `solution` 或 `decision` 文档：

- 原子经验不足以解释背景
- 修复步骤超过两步
- 有多个被否掉的备选方案
- 这个决策以后很可能被重新讨论
