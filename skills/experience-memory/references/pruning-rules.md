# Pruning Rules

## 目标

记忆库不是越大越好，而是越准越好。清理的目的不是删得多，而是减少噪音。

## 清理候选

下面这些条目应进入清理候选：

- 相同规则被记录了多次，只是标题略有差异
- 两条规则互相冲突
- 对应文件路径已经不存在
- 规则依赖的库、接口、目录结构已明显变化
- 一条旧决策已经被新决策替代
- 置信度低且长期没有再次被验证

## 状态处理

优先修改状态，不要默认删除：

- `active`
  - 当前仍然有效
- `stale`
  - 可能过期，需要人工复核
- `superseded`
  - 已被新规则或新决策替代

## 去重策略

如果两条经验同时满足下面条件，可以考虑合并：

- `kind` 相同
- `rule` 语义相同
- `tags` 和 `files` 高度重合
- 时间相近，且没有不同上下文

保留信息更完整、置信度更高、时间更新的一条。

## 冲突处理

如果出现冲突：

1. 先看是否同一项目内的阶段性变化
2. 再看哪条更新、更贴近当前代码
3. 如果属于明确替代关系，把旧条目标记为 `superseded`
4. 如果无法判断，先标记 `stale`，不要直接删除

## 删除原则

默认不自动删除任何记忆文件。

删除必须满足两点：

1. 用户明确确认删除
2. 条目已经被判定为重复、无效或错误记录

在未确认前，只允许：

- 标记状态
- 生成候选清单
- 建议合并方案

## 建议节奏

- 每完成一个较大的修复或功能后，检查一次是否新增噪音
- 每个项目阶段性里程碑后，检查一次重复与冲突
- 在晋升到全局前，先做一次项目内去重
