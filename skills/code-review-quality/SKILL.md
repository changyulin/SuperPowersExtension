# 代码审查与质量

## 概览

使用这个 skill 做高信号代码审查：优先发现行为错误、回归风险、安全问题、可维护性问题、性能隐患和测试缺口，而不是把精力浪费在格式争论上。

目标不是“挑刺”，而是给出具体、可操作、能帮助作者成长的反馈。

## 审查原则

- 先理解意图，再评价实现。
- 先看行为和风险，再看风格和偏好。
- 只报告有证据的问题；不凭空猜测。
- 每条反馈都说明为什么重要，以及建议如何改。
- 在一次审查里给出完整反馈，避免拿到一点信息就零散输出。
- 如果没有发现问题，明确说明没有发现，并补充剩余风险或未验证区域。

## 适用场景

当用户要求“审一下代码”“代码评审”“看下这个改动”“帮我 review”、检查 PR、diff、补丁、重构、bugfix、生成代码或质量风险时使用这个 skill。

如果用户只是要求实现功能、解释代码或运行测试，先完成对应任务；只有当请求包含审查、评审、review、风险检查、质量把关等意图时再进入审查模式。

## 审查流程

1. 先读变更范围：diff、相关文件、测试、配置、接口、数据模型。
2. 重建真实行为路径：输入从哪里来，状态如何变化，输出和副作用落到哪里。
3. 按风险维度审查：正确性、契约与兼容性、安全性、数据与状态、并发与时序、性能与资源、可维护性、测试与可观测性。
4. 把问题按优先级整理，再输出 findings。
5. 在 findings 后补充开放问题、积极评价和剩余风险。

如果变更涉及接口、存储、并发、权限、缓存、迁移或外部依赖，优先扩大上下文，不要只盯着当前几行代码。

## 证据闸门

每个 finding 必须同时满足三点：

- 有可定位证据：文件、行号、diff 片段、测试名、配置键或可复现路径。
- 有具体风险：说明会造成什么错误行为、安全影响、契约破坏、数据状态问题或维护成本。
- 有最小动作：给出作者下一步可以直接执行的修复、补测或澄清方向。

如果缺少任一点，不要把它写成 finding；改放到 `Open Questions` 或 `剩余风险`。

## CHECKPOINT：先澄清再定性

出现以下情况时，先问用户或标成开放问题，不输出确定性 finding：

- 审查目标不清，例如只说“看看”但没有说明是质量、安全、性能还是契约风险。
- 没有可定位变更，且无法从已给文件定位到函数、配置键、接口或测试。
- 结论取决于未给出的业务约束、调用方兼容承诺、数据迁移策略或生产环境开关。

## 上下文扩大触发器

命中下面任一条件时，先补读相关调用链，再下结论：

- API、DTO、事件、Excel/JSON/Markdown 等用户可见输出合同变化。
- 数据库写入、删除、迁移、事务、缓存、唯一约束或状态合并变化。
- 鉴权、授权、租户隔离、令牌传递、日志、外部回调或第三方 API 变化。
- 异步任务、重试、取消、超时、并发锁、队列或后台任务变化。
- SQL 生成/执行、文件路径、模板渲染、反序列化或用户输入拼接变化。

## 优先级

- `阻塞`：必须修复。会导致错误行为、安全漏洞、数据损坏、契约破坏、严重回归，或关键路径缺少必要保护。
- `建议`：应该修复。会明显影响可维护性、可读性、性能、可测试性或未来演进成本。
- `可选`：可以考虑。属于次要改进、局部重构、命名优化、文档补充或更清晰的实现方式。

不要把格式化、个人偏好或已经由 linter/formatter 兜底的事情提升成高优先级问题。

## 输出要求

- 先给 findings，再给总结。
- findings 按严重程度排序。
- 每条 finding 都要包含具体位置，例如文件和行号。
- 每条 finding 都要说明风险、原因和建议动作。
- 当意图不清楚且结论取决于上下文时，用开放问题标注，不要武断定性。
- 如果代码里有明显好的边界、测试或防御式处理，在 findings 后指出来。

优先使用 `references/output-format.md` 里的输出模板。需要系统化排查时，使用 `references/review-checklist.md` 的清单。

## Finding 模板

```text
[优先级] 标题
位置: path/to/file:123

为什么重要:
- 说明错误行为、风险或长期成本

建议:
- 给出最小、可执行的修复方向
```

## 默认审查重点

### 正确性

- 业务逻辑是否真的实现了预期行为。
- 边界条件、空值、异常路径、回滚路径是否覆盖。
- 重构后是否改变了原有语义。

### 契约与兼容性

- API、事件、DTO、数据库字段、配置键或返回结构是否改变。
- 调用方、序列化、文档、测试和示例是否同步。
- 是否破坏向后兼容、用户可见格式或已有集成。

### 安全性

- 输入是否被验证、转义、约束。
- 权限、认证、授权、租户隔离是否正确。
- 是否泄露敏感信息、令牌、内部错误细节。

### 数据与状态

- 写入、删除、迁移、缓存、索引和唯一约束是否保持一致。
- 异常、重试、回滚、补偿和幂等是否覆盖关键路径。
- 是否可能导致重复写入、丢失更新、脏数据或部分成功状态。

### 并发与时序

- 是否存在竞争条件、死锁、乱序、重复消费或后台任务堆积。
- 锁、队列、异步任务、取消和超时处理是否安全。
- 是否依赖不稳定的请求顺序、线程模型或时间窗口。

### 可维护性

- 命名和职责边界是否清晰。
- 是否引入重复逻辑、隐式约定或难以测试的分支。
- 未来改动时，是否容易误用或踩坑。

### 性能

- 是否存在明显的重复计算、无界循环、N+1、过大对象复制或不必要的 IO。
- 是否在高频路径引入了额外锁、网络调用或序列化成本。

### 测试与可观测性

- 关键行为、失败路径和边界条件是否有测试。
- 测试是否真的覆盖风险，而不是只覆盖 happy path。
- 如果没有测试，是否至少指出未验证的高风险区域。
- 日志、指标、trace 和错误信息是否足够定位问题，且不会泄露敏感信息。

## 失败分支与兜底

| 触发条件 | 一线处理 | 仍无法确认时 |
|---|---|---|
| 没有 diff 或行号 | 先要求用户提供 diff，或基于已给文件定位到函数/配置键 | 不编造行号；把判断写到 `Open Questions` |
| 证据只支持“可能” | 查调用方、测试、配置和数据模型补证据 | 写成风险假设，说明需要验证什么 |
| 高风险链路不完整 | 优先追踪入口、持久化、外部副作用和回滚路径 | 标注未验证范围，不给确定结论 |
| 发现多个同类小问题 | 合并成一个模式级 finding | 不拆成多轮低信号意见 |
| 用户只要审查结论 | 只输出 findings 和必要上下文 | 不直接改代码，除非用户随后要求 |

## 不要这样审查

- 不要只盯风格、排版或“我更喜欢另一种写法”。
- 不要把未证实的猜测写成结论。
- 不要在没有上下文时忽略接口契约、数据模型或调用链影响。
- 不要把多个小意见拆成多轮零散输出。
- 不要只批评不指导；给出能执行的下一步。
