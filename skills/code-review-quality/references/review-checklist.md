# 审查清单

按下面顺序审查，优先看高风险区域，不要求机械地逐项朗读。

## 1. 正确性

- 这个改动真的实现了需求吗？
- 输入、状态变化、输出、副作用是否一致？
- 空值、异常、超时、重试、回滚、幂等是否考虑到了？
- 是否存在 off-by-one、条件分支遗漏、默认值误用、顺序错误？

## 2. 契约与兼容性

- 是否修改了 API、事件、DTO、数据库字段、配置键或返回结构？
- 调用方、序列化、反序列化、文档、测试是否同步？
- 是否会破坏向后兼容或已有集成？

## 3. 安全性

- 外部输入是否经过校验、约束、转义或白名单处理？
- 是否存在注入、越权、XSS、路径遍历、敏感数据泄露风险？
- 日志、错误消息、监控事件里是否暴露密钥、令牌、用户隐私或内部实现细节？

## 4. 数据与状态

- 是否可能导致数据丢失、重复写入、脏写或不一致状态？
- 状态更新是否具备事务性或补偿机制？
- 缓存、索引、唯一约束、版本字段是否受影响？

## 5. 并发与时序

- 是否存在竞争条件、死锁、丢失更新、重复消费或乱序问题？
- 异步任务、锁、队列、重试和取消流程是否安全？
- 是否假设了不稳定的时间顺序、线程模型或请求顺序？

## 6. 性能与资源

- 是否引入了明显的 N+1、重复计算、全量扫描或不必要的序列化？
- 是否在热点路径增加了网络往返、磁盘 IO、大对象分配或锁竞争？
- 是否可能造成内存增长、连接泄漏、句柄泄漏或后台任务堆积？

## 7. 可维护性

- 命名和职责边界是否清楚？
- 是否引入了重复逻辑、隐藏约定、过长函数或过深嵌套？
- 是否把复杂度压到了合适的抽象层，而不是散落在调用方？

## 8. 测试与可观测性

- 关键路径、失败路径和边界条件是否有测试？
- 测试是否覆盖真实风险，而不是只有 happy path？
- 日志、指标、trace、错误信息是否足够定位问题？

## 9. 好代码信号

如果看到下面这些点，值得在 findings 后点名表扬：

- 清晰的职责边界
- 明确的失败处理和回滚逻辑
- 有针对性的回归测试
- 防御式编程恰到好处
- 复杂逻辑被提炼成易理解的小单元
