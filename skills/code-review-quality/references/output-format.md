# 输出格式

默认先给 findings，再给开放问题、积极反馈和简短总结。

## 推荐结构

```text
Findings
- [阻塞] 标题
  位置: path/to/file:123
  为什么重要: ...
  建议: ...

- [建议] 标题
  位置: path/to/file:45
  为什么重要: ...
  建议: ...

Open Questions
- 如果某个判断依赖未确认的业务意图或上下文，把问题放这里

Positives
- 点出值得保留的设计、边界、测试或防御式处理

Summary
- 用 1 到 3 句概括整体质量、剩余风险和建议下一步
```

## 书写要求

- findings 按严重程度排序，不要把总结放在前面淹没问题。
- 每条 finding 都要可定位、可解释、可执行。
- 不要只写“这里不好”；说明会造成什么风险。
- 如果没有发现问题，明确写出“未发现阻塞性问题”，并补充未验证区域或测试缺口。
- 如果证据不足，不要硬下结论，改写成开放问题。

## 好的 finding 示例

```text
[阻塞] 错误路径会吞掉事务回滚
位置: src/orders/service.py:87

为什么重要:
- 当前 except 分支记录日志后直接返回，事务没有显式回滚
- 如果上游在同一请求里继续写库，可能留下部分提交状态

建议:
- 在异常分支里显式回滚，或把写操作收敛到统一事务边界内
```

## 不好的 finding 示例

```text
[建议] 这里看起来不太对，最好重构一下
```

问题在于：

- 没有位置
- 没有证据
- 没有风险说明
- 没有可执行建议
